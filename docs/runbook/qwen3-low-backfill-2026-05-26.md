# Qwen3-low Bulk Backfill Runbook (CP488+)

**Target rows**: 491 `video_rich_summaries` rows with `quality_flag='qwen3_low'` (set by PR #752 deploy-side UPDATE).

**Why this exists**: PR #752 hid the broken qwen3-30b-a3b segments from users with a FE guard. PR #754 fixed the new-generation path (Sonnet 4.6 + cron transcript). But the 491 already-broken rows still need fresh content — Sonnet 4.6 via the cron's Track A2 would eventually pick them up if their schema matched, but Track A2 only retries `quality_flag='low'`/`pending`, not `qwen3_low`. So we backfill manually using Mac Mini Claude Code (Opus 4.7).

## Why Mac Mini + Claude Code (and not OpenRouter Sonnet batch)

Cost — the user's Claude Code subscription covers Opus 4.7 inference at $0 marginal. A 491-row OpenRouter Sonnet 4.6 batch would cost ~$25. The user explicitly chose the CC console path.

Quality — the existing 1,273 `claude-code-direct` rows (the highest-quality cohort in production: atoms avg 10.4, sections avg 5.1) were generated by CC console batches. We're reproducing the same pipeline.

## Pre-flight

1. **Confirm PR #752 + PR #753 + PR #754 are all merged.** Otherwise:
   - PR #752 (FE guard) → users will see broken chapters during the backfill window.
   - PR #753 (cron disable) → cron may regenerate `qwen3_low` rows back to qwen3 mid-backfill.
   - PR #754 (Sonnet 4.6 + cron captioner) → new rows could land while you're working.

2. **Local env**: this script uses the production caption extractor (which proxies through Mac Mini when `MAC_MINI_TRANSCRIPT_URL` is set). Run it **from Mac Mini**, not EC2 — EC2 us-west-2 outbound to YouTube is rate-limited.

3. **DATABASE_URL** — read-only is enough for the SELECT; the script does **not** UPDATE.

## Step 1 — Fetch transcripts

```bash
cd ~/cursor/insighta  # or wherever the repo is cloned on Mac Mini
DATABASE_URL="postgresql://..." \
  npx tsx scripts/fetch-transcripts-for-qwen3-backfill.ts \
    --output /tmp/qwen3-low-transcripts.jsonl
```

Expected wall-clock: ~15-30 minutes serial. The script appends one JSON line per video:

```json
{"video_id":"3T75_phv4cA","language":"ko","title":"...","description":"...","channel":"...","transcript":"...","fetched_at":"2026-05-26T..."}
```

On caption-unavailable, the line has `transcript: null` and `error: "..."`. Those rows will be skipped by the regeneration step — their FE guard remains in place.

**Resume on failure**:
```bash
npx tsx scripts/fetch-transcripts-for-qwen3-backfill.ts \
  --output /tmp/qwen3-low-transcripts.jsonl \
  --skip-existing
```

## Step 2 — Regenerate via Claude Code

Open a Mac Mini terminal in the Insighta repo and start a Claude Code session (`claude`). Paste the following starter prompt verbatim:

> Regenerate v2 rich-summary rows for the videos in `/tmp/qwen3-low-transcripts.jsonl`. For each batch of 100, follow this exact contract:
>
> 1. Read 100 lines from the JSONL (skip any with `transcript: null`).
> 2. For each video, produce the layered v2 JSON matching the schema in `src/modules/skills/rich-summary-types.ts` (`RichSummaryV2Layered`). Specifically:
>    - `core.one_liner` (≤20 chars, no quotes), `core.domain` (one of the 9 SSOT slugs), `core.depth_level`, `core.content_type`, `core.target_audience`.
>    - `analysis.core_argument` (2-3 sentences), `analysis.key_concepts` (3-5 items), `analysis.entities` (5-10 typed), `analysis.actionables`, `analysis.mandala_fit` (suggested_goals, relevance_rationale, mandala_relevance_pct), `analysis.bias_signals`, `analysis.prerequisites`.
>    - `lora.qa_pairs` (1-3 level-1 questions sourced from the video).
>    - `segments.sections` (3-8 sections, each with from_sec, to_sec, title, summary, relevance_pct, key_points), `segments.atoms` (5-15 atoms, each with idx, type, text, timestamp_sec, entity_refs).
> 3. **Atoms MUST be sorted by `timestamp_sec` ascending and cover the full video duration**. Sections must tile the timeline without overlap.
> 4. Output language matches the detected language of title/transcript ('ko' or 'en').
> 5. Produce a single bulk UPDATE statement per batch:
>    ```sql
>    UPDATE video_rich_summaries vrs SET
>      core = b.core::jsonb,
>      analysis = b.analysis::jsonb,
>      lora = b.lora::jsonb,
>      segments = b.segments::jsonb,
>      one_liner = b.one_liner,
>      transcript_used = true,
>      template_version = 'v2',
>      quality_flag = 'pass',
>      model = 'claude-code-direct',
>      mandala_relevance_pct = b.mandala_relevance_pct::int,
>      updated_at = now()
>    FROM (VALUES
>      ('video_id_1', 'core_json...', 'analysis_json...', 'lora_json...', 'segments_json...', 'one_liner', 65),
>      ...
>    ) AS b(video_id, core, analysis, lora, segments, one_liner, mandala_relevance_pct)
>    WHERE vrs.video_id = b.video_id;
>    ```
> 6. Save the SQL to `/tmp/qwen3-backfill-batch-{N}.sql`.
> 7. After all batches: confirm the user is ready, then apply via `bash scripts/ssh-connect.sh "docker exec -i insighta-api psql ..."` or have the user run psql directly.

The CC session does the heavy lifting (analysis from transcript). User reviews the output SQL before applying.

## Step 3 — Apply

For each generated batch SQL:

```bash
cat /tmp/qwen3-backfill-batch-1.sql | \
  bash scripts/ssh-connect.sh \
  'docker exec -i insighta-api sh -c "node -e \"const{PrismaClient}=require(\\\"@prisma/client\\\");const p=new PrismaClient();process.stdin.on(\\\"data\\\",async d=>{const r=await p.\\\$executeRawUnsafe(d.toString());console.log(\\\"rows=\\\",r);await p.\\\$disconnect();process.exit(0)})\""'
```

Or simpler — copy the SQL into the Supabase dashboard SQL editor and execute manually. The UPDATE is idempotent (filtered by `video_id`).

## Step 4 — Verify

```bash
# Count remaining qwen3_low rows
echo "SELECT quality_flag, COUNT(*) FROM video_rich_summaries WHERE quality_flag IN ('qwen3_low', 'pass') AND template_version='v2' GROUP BY 1;" | \
  bash scripts/ssh-connect.sh 'docker exec -i insighta-api psql "$DIRECT_URL"'
```

Expected: `qwen3_low` decreases by however many rows had a transcript in step 1; rows without a transcript stay marked `qwen3_low` and remain hidden by PR #752's FE guard until a future captioner attempt succeeds.

## Step 5 — Re-enable the cron

After backfill completes (or partial backfill if some captions are still unavailable), revert PR #753 — `RICH_SUMMARY_V2_CRON_ENABLED=true` again. The cron will now:

- Run `getCaptionExtractor().extractCaptions(...)` first (PR #754).
- Pass the transcript to `generateRichSummaryV2` (PR #754).
- Use `anthropic/claude-sonnet-4-6` for generation (PR #754).

New rows generated by the cron from this point onward should match the claude-code-direct baseline quality.

## Failure modes + rollback

| Symptom | Cause | Fix |
|---|---|---|
| `tsx` fails with `MAC_MINI_TRANSCRIPT_URL not set` | Running on EC2 instead of Mac Mini | Run from Mac Mini; or set `MAC_MINI_TRANSCRIPT_URL` env locally |
| `transcript: null` for >50% of rows | Captioner regression on YouTube side | Pause; investigate; resume with `--skip-existing` after fix |
| UPDATE SQL errors with `jsonb` parse | CC produced malformed JSON | Re-prompt for the specific batch; do not hand-edit the SQL |
| Rows still rendering as "improving" after UPDATE | Browser cache / CDN stale | Hard refresh; check `quality_flag` directly via psql |

## Cost / cadence

- Inference cost: $0 (user's Claude Code subscription).
- Wall-clock: ~2-4 hours for the regeneration step (491 / 100 = 5 batches × ~30 min CC session each).
- Transcript fetch wall-clock: ~15-30 min serial.
- Total: ~half a day's worth of background CC work, no production blast radius.

## Out of scope

- Adding `qwen3_low` to the cron's Track A2 selector. The cron retry uses Sonnet 4.6 (per PR #754); having it pick up qwen3_low rows automatically would work, but at OpenRouter Sonnet cost (~$25 for 491 rows). The user explicitly chose the free CC console path; the cron stays focused on `low`/`pending`.
