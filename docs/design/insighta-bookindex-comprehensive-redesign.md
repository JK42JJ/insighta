# 북인덱스 종합 생성 파이프라인 — 재설계 설계서

> **성격**: 제품 근간 재설계. 노트 뷰 UI 잔손질과 차원이 다른 별도 트랙.
> **위치**: 메모리 ARCHITECTURE-bookindex.md(repo SSOT) §1 ③④⑤⑥의 실제 구현 격차를 닫는 작업.
> **단계**: 본 문서 = 설계(2단계 산출). 이후 단계별 구현 핸드오프로 쪼갬.
> **작성 맥락**: CP500+ 2026-06-24. 55카드 만다라에서 북인덱스가 7~8 카드로만 생성됨을 James가 발견 → 감사로 "단편적 v2-반영" 확정 → 재설계.

---

## 0. 왜 이게 근간인가

비전(vision-backbone §4): **"탄탄해진 만다라로 북인덱스를 만들면, 그것이 한 권의 책의 목차이자 본문이 된다."** 북인덱스는 fan-out(슬라이드덱·뉴스레터·챗봇·KG)의 **베이스라인**이다. 북인덱스가 얕으면 그 위 모든 콘텐츠가 얕다. 여기가 무너지면 제품 전체가 무너진다.

그런데 감사(2026-06-24)가 코드로 확정한 현실:

> **현 파이프라인 = "단편적 v2-반영" (종합적 북인덱스 아님).**

7~8/55 카드로만 북인덱스가 구성되고, 그것도 "우연히 하트된 카드"다. 이건 "개판"이 아니라 — **설계(§1)는 옳은데 구현이 §1④·⑥에 안 닿은 것.** 한 군데(§1④)가 빠져서 전체가 무너졌다.

## 1. 감사 확정 사실 (코드 증거)

| 단계 | 설계(§1) | 현 코드 | 위치 |
|---|---|---|---|
| ③ 게이트(rel≥40) | 통과/탈락 | ✅ | `passesBookGate` (fill-book:95) |
| ④ **통과 카드 v2화** | 게이트 통과분 v2 보장 | ❌ **없음** | v2 필터가 게이트 **前**(93→95). 통과카드 v2 enqueue 루프 0 |
| ⑤ 내용차원 재배치 | **셀내 종합 + 책차원 중복제거·순서재배치** | ⚠️ **반쪽** | 셀내 종합 ✅ `synthesizeCellTopics`(fill-book:112-128) / **cross-cell 중복제거·책 순서재배치 ❌** |
| ⑥ 인리치/팩트체크 | 자료보강·검증 | ❌ **코드 부재** | grep 0 (메모 "4/3 설계有 코드無") |

> **⑤ 정정 (James 지적):** 당초 감사에서 ⑤를 "✅"로 줬으나 부정확. 현 `synthesizeCellTopics`는 **셀 단위로만** atom을 토픽화한다 — 셀 안에서만 종합하고, **셀을 가로지르는 중복 제거도, 책 차원의 순서 재배치도 안 한다.** 즉 셀내 종합 ✅ / 책차원 ❌ = **반쪽.** §4.5 참조.

**근본 원인 (한 줄):** 순서가 뒤집혀 있다 — `v2 필터(93) → 게이트(95)`. v2 안 된 카드는 게이트에 닿기도 전에 `if(!v2) continue`로 스킵된다. 그리고 **게이트 통과 카드를 v2화하는 단계(§1④)가 통째로 없다.** v2는 Heart on-demand로만 생긴다.

**대상 정의는 옳다:** 북인덱스 대상 = rel≥40 통과 카드 (전부 아님, `BOOK_GATE_MIN_RELEVANCE=40`). 합의된 부분. 문제는 그 **대상을 채우는 메커니즘(§1④)이 비어 있다**는 것. 대상 정의 ≠ 대상 충족.

**부수 확정:**
- "전체 카드 v2 확보" 장치 = 없음 (Heart on-demand / 위저드 precompute만).
- #958 트리거는 "v2된 것 반영"만, "전체 v2화" 아님.
- v2=0 셀 → 빈 챕터(제목만) 렌더 = 데이터 공백(디자인 아님).
- 진행도 상태값(done/total, job progress) = 없음 → 스피너 데이터원 부재.

## 2. 설계 결정 — B (lazy, book-fill 시점 §1④ 복원) 확정

James 결정: **B**. book-fill이 돌 때 게이트 통과했는데 v2 안 된 카드를 그 자리에서 v2 enqueue → 완료되면 #958 트리거로 재반영.

**근거:**
1. §1④ 설계가 원래 이거다(게이트 → 통과분 v2 → 종합). 순서만 바로잡으면 됨.
2. eager(A)는 안 볼 수도 있는 만다라까지 Sonnet 비용을 선지출. lazy는 실제 book 트리거분만.
3. 진행도 스피너가 자연히 붙는다 — book-fill이 "통과 N개 중 v2 M개 완료" 상태를 갖게 되므로 그 done/total이 스피너 데이터원(감사 5번 부재분이 여기서 생김).

**트레이드오프(수용):** 첫 book은 불완전 → v2 완료되며 점진 충족(현 7→8 현상이 "전체 통과분까지" 확장). 사용자에겐 스피너로 진행 가시화(§5).

## 3. 종합 파이프라인 — 목표 형상

James 정의 "종합적 북인덱스" = 5단계 완주. 현 구현 대비 격차를 단계별로:

```
[수집] 만다라 전체 카드
   ↓
③ [게이트] rel≥40 통과분 선별                          ✅ 있음 (순서만 교정)
   ↓
④ [통과분 v2 확보] 통과했는데 v2 없는 카드 → v2 enqueue  ❌ 신규 (B, 본 설계 1단)
   ↓  (v2 완료 대기 — #958 트리거로 재진입)
⑤ [구조 재구성] 셀내 토픽종합 + cross-cell 중복제거 + 책 순서재배치   ⚠️ 반쪽 (셀내만, 책차원 신규)
   ↓
⑥ [완성도] 인리치 → 자료보강 → 팩트체크                  ❌ 신규 (skill, 본 설계 2단)
   ↓
[북인덱스] = fan-out 베이스라인 (슬라이드/뉴스레터/챗봇/KG)
```

핵심: **④가 ⑤·⑥의 선결.** 통과 카드가 다 v2돼야 ⑤(중복제거·재배치)·⑥(인리치·팩트체크)이 전체를 대상으로 돈다. 그리고 **⑤(중복제거)가 ⑥(인리치)의 선결** — 중복 그대로 두고 인리치하면 같은 걸 여러 번 입체화하는 낭비. **순서 = ④ → ⑤ → ⑥.**

## 4. §1④ 복원 — 통과분 v2 확보 (1단, 작은 안)

가장 작고 즉효인 수리. 순서 교정 + enqueue 루프.

**현 (틀림):**
```
카드수집 → if(!v2) continue [93] → passesBookGate [95] → 토픽종합
         ^^^^^^^^^^^^^^^^^^^^ v2 없으면 게이트도 못 봄
```

**목표:**
```
카드수집 → passesBookGate [게이트 먼저] → 통과분 분기:
              ├─ v2 있음 → 토픽종합 진입
              └─ v2 없음 → enqueueEnrichRichSummary(card)  ← 신규 루프
                          + book 상태 "v2 대기 N건" 기록
         → (v2 완료 시 #958 트리거로 book-fill 재진입 → 점진 충족)
```

**불변식(신규):**
```
[INV-BOOK-COVERAGE] 북인덱스는 게이트 통과 카드 전체를 대상으로 한다.
  통과했으나 v2 없는 카드는 누락이 아니라 "v2 대기"로 표시되고,
  v2 enqueue가 보장된다.
  강제: (현) ❌ 무방비 (v2된 것만 우연히 반영)
  목표: ✅ 코드 (게이트 통과분 v2 enqueue 루프 + 대기 상태)
```

**경계:** 이 1단은 §1⑥(인리치/팩트체크) 없이도 독립 배포 가능. "통과분 전체가 book에 들어오게" 만드는 것까지. ⑥은 그 위에 얹는 다음 단.

**측정 선결(구현 전 CC 확인):** 전형 만다라에서 게이트 통과 카드 수 vs 현 v2 카드 수. 55카드 만다라면 통과분이 몇 개인지(40? 30?) — 그게 v2 enqueue 폭이자 비용 base(§7).

## 4.5 §1⑤ 책차원 재구성 — 중복제거 + 순서재배치 (꿀빨기의 핵심)

**James 원칙:** 북인덱스를 만드는 과정은 **'꿀빠는' 과정**이다 — 여러 영상에서 나온 파편을 흡수하되, 중복은 버리고, 종합적 구조에 맞는 섹션 순서로 재배치한다. 이게 빠지면 북인덱스는 "책"이 아니라 "영상 모음"이 된다.

**현 격차:** `synthesizeCellTopics`는 셀 단위로만 종합한다. 셀을 가로지르는 중복도, 책의 읽기 순서도 손대지 않는다. 그래서:
- **중복 잔존:** 영상 A·B·C가 다 "클로드 코드 설치"를 다루면 → 북인덱스에 같은 내용 3번. 꿀빨기의 반대 — 파편을 흡수 못 하고 그대로 쌓음.
- **순서 부재:** 셀 순서(만다라 공간 배치)는 분류지 읽기 흐름이 아니다. 책은 전제→기초→실전→심화의 **논리 흐름**이어야 한다. 셀 순서 ≠ 책 순서.

**목표 — ④ 후, ⑥ 전에 삽입되는 두 작업:**

1. **cross-cell 중복 제거:** 통과분 v2 atom 전체에서 의미적으로 겹치는 것 식별·병합. 같은 개념이 여러 영상에 나오면 → 가장 좋은 설명 하나로 통합 + 출처는 다중 참조(`seg_ref` 다중). 꿀빨기 = 파편 흡수, 중복은 버림.

2. **책 차원 순서 재배치:** 셀 순서가 아니라 북인덱스 성격에 맞는 논리 흐름으로 챕터·섹션 재배열. 만다라의 공간 분류를 책의 시간(읽기) 순서로 변환.

**불변식(신규):**
```
[INV-BOOK-DEDUP] 북인덱스는 cross-cell 중복을 병합한다. 같은 개념의
  중복 atom은 하나로 통합되고 출처만 다중 참조로 남는다.
[INV-BOOK-ORDER] 북인덱스 섹션 순서는 셀 순서가 아니라 책의 논리 흐름
  (전제→기초→실전→심화 등)을 따른다.
  강제: (현) ❌ 무방비 (셀 순서 그대로, 중복 그대로)
  목표: ✅ 코드 (cross-cell dedup + 책순서 재배치)
```

**순서 결정 (중요):** 중복 제거는 **⑥(인리치) 전**에 와야 한다. 중복을 그대로 두고 인리치하면 같은 걸 여러 번 입체화하는 비용 낭비. **④(통과분 v2) → ⑤(중복제거+순서) → ⑥(인리치/팩트체크).**

**구현 위치:** ⑤가 `synthesizeCellTopics` 다음에 cross-cell 층을 추가하는 형태인지, 별도 단계인지 — 구현 시 측정으로 결정(6.3과 함께). 중복 판정은 의미 기반(임베딩 유사도 / LLM)이라 비용·정확도 trade 존재 → 2단 착수 시 좁힘.

**단 배치:** ⑤ 책차원 재구성은 **2단(§1⑥과 함께)** 에 둔다 — 둘 다 "통과분 전체 v2 확보(1단)" 위에서만 의미가 있고, 중복제거·인리치·팩트체크가 한 묶음의 "완성도" 작업이기 때문. 1단은 "통과분이 다 들어옴"까지, 2단이 "들어온 걸 책으로 정제".

## 5. 진행도 상태 + 좌측 스피너 (1단에 동반)

감사 5번: 진행도 상태값 부재. §1④ 작업이 이걸 만든다.

**데이터원:** book-fill이 갖게 될 상태 — `{ gate_passed: N, v2_done: M, v2_pending: N-M }`. job progress로 노출.

**좌측 만다라 스피너 (James 요구: 심플하지만 직관적):**
- 위치: 좌측 만다라 리스트 아이템 (생성/갱신 중인 만다라).
- 표현: `v2_pending > 0`이면 진행 인디케이터. done/total 비율 또는 미니멀 스피너.
- 상태: "준비 중"(현 슬라이드덱 "준비중" 패턴 재사용 — 이미지2의 ⋯메뉴 `준비중` 라벨과 일관).
- 완료: `v2_pending == 0` → 인디케이터 제거.

**원칙:** 디자인은 단순(스피너/비율 한 개), 의미는 명확("이 만다라 북인덱스 채워지는 중"). 빈 섹션이 "버그처럼" 보이던 걸 "진행 중"으로 정직하게 바꾼다.

**빈 섹션 처리 (동반):** v2 대기 중인 챕터는 "제목만 빈 채"가 아니라 "준비 중" 표시. 데이터 공백을 진행 상태로 대체.

## 6. §1⑥ 완성도 skill — 인리치 / 자료보강 / 팩트체크 (2단, 큰 안)

④로 "전체 통과분 v2 확보"가 되면, 그 위에 ⑥(완성도)을 얹는다. 이게 "강력한 skill"(James)이고, 메모리 skill 3-Layer(SkillRegistry / Temporal / VOC) 위에 오케스트레이션된다.

### 6.1 skill 단계 (순차)

```
[중복제거+순서]  ⑤ cross-cell dedup + 책 순서재배치 (§4.5)  ← 인리치 前 선결
   ↓
[인리치]  v2 atom 묶음 → 챕터 내 사실·개념 정리, 누락 토픽 식별
   ↓
[자료보강] 부족·얕은 부분 → 외부 리서치로 입체화 (보강 소스 부착)
   ↓
[팩트체크] 사실 오류·논란 여지 → 체크/보완, 신뢰도 마킹
   ↓
완성된 챕터 (= 북인덱스 베이스라인)
```

> ⑤(중복제거)를 인리치 앞에 두는 이유: 중복 그대로 인리치하면 같은 걸 여러 번 입체화 = 비용 낭비. 꿀빨기(파편 흡수·중복 버림)가 먼저, 입체화가 나중.

### 6.2 3-Layer 매핑 (메모리 skill 아키텍처)

| Layer | 역할 | 북인덱스 ⑥ 적용 |
|---|---|---|
| A — SkillRegistry | SkillInterface 계약, ADD/REMOVE | `book-enrich` skill 등록 (인리치/보강/팩트체크 3 activity) |
| B — Temporal | 실행 보장, per-skill task queue 격리, retry | book-enrich 전용 큐. 3단계 = 3 activity 순차. 장애 격리(팩트체크 실패가 인리치 롤백 안 함) |
| C — VOC | admin-driven human-in-loop | (해당 없음 — 시스템 트리거. VOC는 사용자 요청 스킬용) |

**트리거:** event — 만다라 통과분 v2 완료(`v2_pending → 0`) 시 book-enrich 실행. = §1④ 완료가 §1⑥의 트리거.

### 6.3 결정 필요 사항 (2단 착수 전)

- **자료보강 소스/도구:** 외부 리서치 = web_search 류? 어느 모델? (비용·정확도 trade)

### 6.4 `/config` 구문 하이라이트 (⑥ 인리치 산출물)

이미지5: 본문 `/config` 같은 코드성 구문이 하이라이트 안 됨. 이건 표시 레이어 버그가 아니라 **인리치 단계에서 "이건 코드칩"이라고 마킹해야 하는 데이터 문제.** 시안(insighta-note-redesign.html)의 code칩(mono `.82em` `#dcc69e`)은 마크업이 있어야 렌더된다 → 인리치가 atom 텍스트에서 코드 토큰을 식별·마킹 → 표시 레이어는 마킹된 것만 칩으로 렌더. ⑥ 인리치 산출 스키마에 포함.
